# 豆包播客自动化下载与 Obsidian 绑定 —— 任务总结

> 一句话：从豆包 AI 的"根据文档生成播客"功能中，全自动提取音频、压缩为 MP3，并绑定回 Obsidian 知识库的 Markdown 笔记。

---

## 1. 实际解决了什么问题

### 1.1 用户痛点

豆包 AI 提供"根据文档生成播客"功能，可以将 PDF 转换为 AI 生成的播客音频（双人对话形式）。用户在日常使用中积累了大量播客，但面临以下问题：

| 痛点 | 说明 |
|------|------|
| 无法批量导出 | 豆包网页没有提供批量下载功能，每个播客需要手动点击下载 |
| 文件名无意义 | 下载后的文件名是 `AI播客_123456.wav`，与原始 PDF 无关 |
| 音频与笔记脱节 | Obsidian 中已有对应 PDF 的 Markdown 笔记，但播客音频与笔记之间没有关联 |
| 文件体积过大 | WAV 文件平均 120MB，不适合直接存储在 Obsidian 附件中 |

### 1.2 解决方案

```
豆包聊天记录 URL
    ↓ 扫描页面所有播客
PDF 文件名 + 音频下载链接
    ↓ 逐个下载为 WAV
原始文件名命名 + WAV
    ↓ FFmpeg 压缩
MP3（体积缩小 79%）
    ↓ 搜索对应 Markdown
在笔记中插入 WikiLink 链接
```

## 3. 技术路径快照

### 3.1 技术选型

| 组件 | 选择 | 原因 |
|------|------|------|
| 浏览器自动化 | Playwright | 原生支持 `expect_download` 和 `expect_file_chooser` |
| 编码/压缩 | FFmpeg | `-q:a 2` VBR 模式，体积缩小 79% |
| 脚本语言 | Python 3.14 | Playwright 的异步 API 原生支持 `async/await` |
| 状态持久化 | JSON 文件 | 无需数据库，简单可靠 |
| 依赖安装 | `pip install -r requirements.txt` | 标准化依赖管理 |

### 3.2 关键技术细节

1. **虚拟列表处理**：豆包使用 CSS Transform 虚拟列表，不在视口内的元素被完全移除。解决方案是：`createTreeWalker` 遍历文本节点 + 滚动触发重新渲染。

2. **登录态管理**：使用 `context.storage_state()` 保存 Cookie/LocalStorage 到 `doubao_state.json`，后续运行自动加载。

3. **精确绑定策略**：以 PDF 文件名为锚点，每个播客卡片上方都有固定文本"我将根据 xxx.pdf 的内容为你生成播客"，以此精确定位。

4. **断点续传**：`upload_progress.json` 记录已处理的 PDF，重启后自动跳过已完成的。

5. **PDF 上传自动化**：使用 `expect_file_chooser` 上下文管理器捕获文件选择弹窗，配合真实鼠标序列触发按钮点击。

## 4. 关键卡点与解决方案

### 4.1 虚拟列表元素找不到

**症状**：Playwright 的 `locator("text=xxx.pdf")` 返回 count=0，但截图中明明能看到。  
**根因**：虚拟列表只渲染视口附近的元素。  
**解决**：`document.createTreeWalker` 遍历文本节点 + `scrollTop` 修改触发 React 重新渲染。

### 4.2 文件名截断 Bug

**症状**：`多智能体协作调查：Agent 到底该怎么分工.pdf` 被命名为 `到底该怎么分工.wav`。  
**根因**：正则 `r'([^\\/:*?"<>|\s]+\.pdf)'` 排除了空格，`Agent ` 中的空格截断了匹配。  
**解决**：扫描器正则去掉 `\s` 限制，改为 `[^\\/:*?"<>|]+\.pdf`。

### 4.3 下载按钮误点"下载电脑版"

**症状**：点击下载后弹出了豆包客户端下载页面。  
**根因**：`button:has-text("下载")` 匹配到了右上角的"下载电脑版"按钮。  
**解决**：改为查找 `class*="download"` 的元素，并限制在 `data-plugin-identifier*="receive-podcast-content"` 的播客卡片内部。

### 4.4 PermissionError（文件被占用）

**症状**：`shutil.move()` 时文件被浏览器占用。  
**解决**：`shutil.move()` 前增加重试循环，等待浏览器释放文件句柄。

### 4.5 "+" 按钮点击不触发菜单

**症状**：`element.click()` 返回成功，但附件菜单未弹出。  
**根因**：豆包的 "+" 按钮监听 `pointerdown`/`pointerup` 事件，`element.click()` 只派发 `click` 事件。  
**解决**：改用真实鼠标序列 `mouse.move` → `mouse.down` → `mouse.up`。

### 4.6 filechooser 捕获超时

**症状**：`expect_file_chooser` 超时。  
**根因**：filechooser 事件在监听开始之前就已经弹出了。  
**解决**：将触发 filechooser 的操作放在 `expect_file_chooser` 上下文管理器**内部**。

### 4.7 "生成播客"误点 sidebar

**症状**：点击"生成播客"后进入了错误的聊天窗口。  
**根因**：搜索范围是整个页面，匹配到了左侧历史列表中的"根据文档生成播客"。  
**解决**："以 PDF 为中心"的定位策略，只在对应 PDF 消息卡片内查找按钮。

## 5. 可复用经验

### 5.1 Playwright 最佳实践

1. **下载监听**：`expect_download()` 上下文管理器必须在点击下载按钮**之前**进入
2. **File Chooser**：`expect_file_chooser()` 上下文管理器必须包含触发操作
3. **真实鼠标**：对于自定义事件监听的按钮，使用 `mouse.move/down/up` 代替 `element.click()`
4. **滚动触发虚拟列表**：修改 `scrollTop` 而非 `scrollTo`，更容易触发 React 重新渲染

### 5.2 跨环境迁移经验

1. **编码问题**：Windows 终端默认 GBK，`subprocess.run(text=True)` 必须指定 `encoding='utf-8', errors='ignore'`
2. **路径计算**：永远使用 `os.path.relpath()` 而非硬编码相对路径
3. **依赖隔离**：使用 `venv` 或 `conda`，并锁定依赖版本

### 5.3 批处理设计原则

1. **分 batch 执行**：避免单任务超时
2. **断点续传**：每批结束后持久化状态
3. **幂等性**：已处理的文件自动跳过，不报错
4. **重试机制**：文件占用等临时问题应自动重试

### 5.4 正则表达式教训

**不要**用 `\s` 排除空格来匹配文件名：
- `[^\\/:*?"<>|\s]+\.pdf` ❌ —— 会截断含空格的 PDF 名称
- `[^\\/:*?"<>|]+\.pdf` ✅ —— 正确匹配含空格的 PDF 名称

## 6. 后续优化方向

### 6.1 短期优化

1. **单浏览器会话**：目前扫描和下载分两次打开浏览器，可优化为一次会话内先扫描再下载
2. **GUI 界面**：用 `tkinter` 或 `gradio` 做简单的图形界面，显示进度条
3. **更多格式支持**：目前只支持 MP3，可增加 M4A（AAC）格式选项

### 6.2 中期优化

1. **并发下载**：Playwright 支持多个 `page` 共享一个 `context`，理论上可同时操作多个标签页下载
2. **Headless 模式**：稳定后改为无头模式，节省资源
3. **CI/CD 集成**：GitHub Actions 定时执行，自动同步新播客

### 6.3 长期优化

1. **API 化**：将核心功能封装为 REST API，支持远程调用
2. **多平台支持**：除豆包外，支持 ChatGPT、Claude 等平台的播客导出
3. **智能匹配**：利用 LLM 实现音频内容与 Markdown 内容的语义匹配，而非仅基于文件名

核心亮点：
- **全自动**：从 PDF 上传到播客生成，从音频下载到笔记绑定，全程无需人工干预

- **跨环境兼容**：解决了 Windows GBK 编码、路径计算等跨平台问题
- **精确匹配**：以 PDF 文件名为锚点，确保音频与笔记的准确关联
