## 一、任务概述

### 1.1 背景与动机

用户在日常使用豆包 AI 的"根据文档生成播客"功能时，积累了大量由 PDF 生成的播客音频。这些音频散落在豆包网页中，存在以下痛点：

- **无批量导出**：豆包网页没有提供批量下载功能，每个播客需要手动点击下载
- **命名混乱**：下载后的默认文件名（如 `AI播客_123456.wav`）与原始 PDF 无关，无法在文件系统中识别内容
- **知识孤岛**：Obsidian 知识库中已有对应 PDF 的 Markdown 笔记，但播客音频与笔记之间没有关联，形成了"有笔记无音频、有音频无笔记"的双轨割裂

### 1.2 任务目标

给定一个豆包聊天记录 URL，全自动完成以下流程：

```
扫描页面所有播客 -> 按原始PDF名称下载 -> WAV压缩为MP3 -> 删除WAV -> 绑定到对应Markdown
```

### 1.3 约束条件

- 豆包是 SPA（单页应用），使用虚拟列表，只渲染视口内的 DOM 元素
- 需要登录态才能访问聊天记录
- 音频文件体积大（WAV 平均 120MB）
- 最终交付物必须嵌入 Obsidian 的 WikiLink 语法

## 四、KPT 复盘

### 4.1 Keep（保持）

**1. Playwright 的 expect_download 事件监听**
Playwright 原生提供的下载事件监听比 Selenium 的第三方代理方案更可靠，是整个流程的基石。

**2. 以 PDF 文件名为锚点的精确绑定**
每个播客卡片上方都有固定文本"我将根据 xxx.pdf 的内容为你生成播客"。PDF 文件名在整个页面中唯一，比标题或 class 选择器更稳定。

**3. 登录态持久化（storage_state）**
将 Cookie 和 LocalStorage 保存为 JSON，下次自动加载，将人工干预从"每次"降低为"首次"。

**4. 断点续传机制**
下载前检查文件是否已存在，分 batch 重新下载时节省大量时间。

**5. 分 batch 并行下载**
将 28 个待下载文件分为 4 个 batch 并行执行，总耗时从预估 60+ 分钟缩短到 30-40 分钟。

### 4.2 Problem（问题）

**1. 虚拟列表导致 DOM 元素"消失"**
豆包使用 CSS Transform 虚拟列表，只有视口附近的元素被渲染。Playwright 的 locator 在元素不在视口内时返回 count=0。

**2. Windows 终端 GBK 编码导致中文乱码**
Git Bash 默认 GBK 编码，Python 输出 UTF-8 中文时显示为乱码。更严重的是，subprocess.run 捕获 FFmpeg 输出时因非 ASCII 字符抛出 UnicodeDecodeError。

**3. 浏览器下载文件被占用（PermissionError）**
shutil.move 移动下载文件时，浏览器尚未释放文件句柄。

**4. 后台任务心跳过期**
单任务超时 1800 秒，但后台 worker 在 14 分钟时因 heartbeat 过期中断。

### 4.3 Try（尝试）

| Try 项 | 具体行动 | 验证标准 |
|--------|---------|---------|
| 单会话扫描+下载 | 同一个浏览器上下文中先扫描再下载 | 总耗时减少 30% |
| Headless 模式 | headless=False 改为 True | CPU 使用率降低 20% |
| 进度条可视化 | 用 tqdm 或 rich 显示下载进度 | 用户可实时看到进度 |
| 自动重试失败项 | 下载失败时自动重试 3 次 | 失败率降低 |
| MD 文件缺失预警 | 扫描阶段检查对应 Markdown 是否存在 | 绑定前可知缺几个 |

## 五、5Why 根因分析

### 问题：task_guide_sec01 为什么连续 3 次下载失败？

**Why 1**：为什么下载被取消？
-> 豆包服务端在 download.save_as() 阶段主动返回了 canceled 状态。

**Why 2**：为什么服务端会取消？
-> 推测该播客的音频文件可能处于"生成中"或"已过期"状态，服务端拒绝了流推送。

**Why 3**：为什么这个播客会处于异常状态？
-> task_guide_sec01 是较早生成的播客（位于聊天记录上方），可能因虚拟列表长期未渲染导致音频资源被服务端回收。

**Why 4**：为什么服务端会回收音频资源？
-> 豆包的播客音频可能是临时生成的 blob URL，有一定的有效期。长时间未被访问的音频可能被清理。

**Why 5**：为什么脚本没有检测到音频已过期？
-> 页面 UI 上该播客仍然显示为"可播放"状态（有播放按钮和时长），但下载链接已失效。UI 状态与服务端状态不同步。

**根因**：豆包播客的"可播放"UI 状态与"可下载"服务端状态存在不一致，脚本无法通过前端 DOM 判断音频是否已过期。

**对策**：
1. 下载失败时自动标记，提示用户手动验证
2. 扫描阶段额外检查播放按钮的响应状态
3. 标记为"可能过期"的播客优先下载

## 六、4L 反思

### 6.1 Liked（喜欢）

- 用户在整个过程中保持了清晰的指令和及时的反馈，特别是在发现"下载错了"时立即指出问题，避免了错误扩散
- Playwright 的 API 设计优雅，特别是 expect_download 的上下文管理器，让下载监听变得极其简洁
- 分 batch 并行策略在第三次尝试时证明有效，体现了"小步快跑、快速迭代"的敏捷思维

### 6.2 Learned（学到）

- **虚拟列表是现代 SPA 的常态**：React/Vue 的大型列表几乎都用虚拟滚动，任何浏览器自动化工具在处理这类页面时都必须考虑"元素不在 DOM 中"的情况
- **TreeWalker 是遍历文本节点的利器**：document.createTreeWalker 可以穿透 shadow DOM 和复杂嵌套，比 querySelectorAll 更适合文本搜索场景
- **Windows 终端编码是隐形杀手**：GBK 编码问题不仅导致乱码，还会让 subprocess 直接崩溃。关键结果必须保存为文件，不能依赖终端输出
- **断点续传是长任务的生命线**：没有断点续传，后台任务中断后必须从头开始，时间成本不可接受

### 6.3 Lacked（缺乏）

- **缺少对豆包页面结构的预先调研**：如果在编码前先做一次完整的 DOM 结构分析（而不是边做边探），可以更早发现虚拟列表的问题
- **缺少异常重试机制**：下载失败、文件被占用等情况没有内置自动重试，导致需要手动补下
- **缺少进度可视化**：用户只能等待后台任务完成，无法实时看到"第 23/52 个"的进度
- **缺少对 Obsidian 库的前置检查**：如果扫描阶段就发现某播客没有对应 Markdown，可以提前告知用户

### 6.4 Longed for（期望）

- 希望有一个简单的 GUI 界面（如 tkinter），显示下载进度和成功率
- 希望支持多个豆包 URL 的队列批量处理（用户可能有多个聊天记录需要处理）
- 希望支持增量同步：只下载"上次同步后新生成"的播客
- 希望在压缩 MP3 后自动生成音频封面或元数据标签（ID3），让 Obsidian 的音频播放器显示标题

## 八、设计原理摘要

### 8.1 两段式架构

```
浏览器层（Playwright）          本地层（Python）
    |                              |
    |- 打开豆包 SPA               |- 监听下载事件
    |- 滚动虚拟列表               |- 重命名文件
    |- 点击下载按钮               |- FFmpeg 压缩
    |- 捕获下载事件               |- 修改 Markdown
```

### 8.2 虚拟列表处理

- 通过修改滚动容器的 scrollTop 触发 React 重新渲染
- 用 document.createTreeWalker 遍历文本节点，向上追溯到播客卡片
- 在卡片内点击下载按钮

### 8.3 精确绑定

- 以 PDF 文件名为锚点
- 在浏览器内 TreeWalker 搜索 -> 向上追溯到 data-plugin-identifier -> 点击下载
- 下载后用 PDF 名称（去掉 .pdf）作为文件名
